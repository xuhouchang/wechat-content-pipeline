# WeChat Content Platform Redesign

**Date:** 2026-06-03

**Status:** Draft for review

## Goal

Replace the current script mesh with a pure code-driven content platform that:

- separates data collection from business generation
- provides a single scheduler entrypoint with durable task state
- lets `daily_article` and `case_study` share the same material platform
- reduces repeated topics, repeated source reuse, and hidden execution failures
- keeps storage file-based and automatically cleans data older than 30 days

## Why the current design is unstable

The current repository already moved part of the logic out of OpenClaw, but it still behaves like an agent-era system:

- orchestration is split across `run_all.sh`, `run_daily_article.sh`, `poll_and_save.py`, `write_article.py`, `wechat_publish.py`
- task handoff depends on temp files and polling rather than explicit stage completion
- material state is duplicated across `tmp/`, `reports/`, `all_urls.tsv`, `_recent_topics.json`, `.state/`
- URL status updates happen in multiple places with slightly different logic
- business logic reads raw material storage directly instead of consuming a stable curated dataset
- article diversity is controlled by heuristics bolted onto the writer, not by the shared material layer
- editorial filtering is too broad, so topics that are AI-related but not truly about enterprise organizational change still enter the writing pool

This produces exactly the failures the user described:

- cron wakes up but a downstream stage silently fails or partially completes
- a stage writes output but no later stage treats it as authoritative
- repeated materials leak through because dedup is URL-level and local to one script
- repeated topics leak through because topic diversity is not part of dataset planning

## Recommended architecture

Use a file-based content platform with six explicit layers:

1. `ingest`
   Raw fetch outputs from RSS, blogs, consulting, search, and case discovery.
2. `normalize`
   Canonicalized material records with normalized URL, source, date, title, content hash, language, metadata, and extraction quality.
3. `curate`
   Cleaned and filtered material records with relevance verdicts, tags, cluster IDs, dedup decisions, and evidence quality.
4. `datasets`
   Business-ready dataset snapshots generated for a date or run window.
5. `jobs`
   Durable run manifests, per-step status, retries, logs, and artifacts.
6. `outputs`
   Final business outputs such as article drafts, case-study drafts, images, and publish results.

The business layer must never read `ingest` or `normalize` directly. It only reads datasets generated by the platform.

## Scheduler and job model

Cron should only wake a single entrypoint, for example:

```bash
python3 platform_cli.py run collect-daily --date 2026-06-03
python3 platform_cli.py run article-daily --date 2026-06-03
python3 platform_cli.py run case-daily --date 2026-06-03
python3 platform_cli.py run cleanup --date 2026-06-03
```

Inside the platform, each job writes a durable `job.json` like this:

```json
{
  "job_id": "article-daily_2026-06-03_001",
  "job_type": "article-daily",
  "date": "2026-06-03",
  "status": "running",
  "started_at": "2026-06-03T06:00:00+08:00",
  "finished_at": null,
  "attempt": 1,
  "steps": [
    {"name": "build_dataset", "status": "success"},
    {"name": "pick_topic", "status": "running"},
    {"name": "write_article", "status": "pending"},
    {"name": "publish_draft", "status": "pending"}
  ],
  "artifacts": {
    "dataset": "platform/datasets/2026-06-03/article_pool.json"
  },
  "error": null
}
```

Rules:

- every step is idempotent
- every step has one explicit input and one explicit output artifact
- rerun resumes from the first failed or missing step
- a later step is never allowed to reconstruct a missing earlier artifact silently
- logs live under the same job folder as structured JSONL plus human-readable log text

This replaces shell background polling and hidden waits.

## Shared platform pipeline

The shared pipeline should be:

1. `collect_sources`
   Pull raw source pages and feed results into `platform/ingest/raw/<date>/...`
2. `normalize_materials`
   Convert raw outputs into canonical material records, normalize URLs, strip boilerplate, compute hashes
3. `curate_materials`
   Run relevance filter, quality checks, language guardrails, tagger, and dedup clustering
4. `build_datasets`
   Produce business-facing dataset snapshots for article and case use
5. `run_business_pipeline`
   The article or case flow consumes a dataset snapshot, not raw materials
6. `record_usage`
   Mark which canonical materials and clusters were used by which business output
7. `cleanup`
   Remove old ingest and intermediate files older than 30 days, while keeping compact state summaries

## Business pipeline split

`daily_article` and `case_study` should share layers 1 to 4 and diverge only at layer 5.

### Daily article flow

Purpose:

- create one narrative-driven WeChat article from a curated pool
- optimize for insight, readability, and novelty

Input:

- `platform/datasets/<date>/article_pool.json`

Own logic:

- topic planning
- novelty scoring
- material selection
- article writing
- image search and embedding
- WeChat draft publish

Output:

- `platform/outputs/daily-article/<date-slug>/`

### Case study flow

Purpose:

- create one high-density operational case breakdown
- optimize for details, workflow clarity, and reusable operational judgment

Input:

- `platform/datasets/<date>/case_pool.json`

Own logic:

- case candidate ranking
- evidence merging
- cross-source validation
- case writing
- image search and embedding
- WeChat draft publish

Output:

- `platform/outputs/case-study/<date-slug>/`

The important boundary is that neither flow should implement its own source discovery, URL dedup, tagging store, or material usage registry.

## Material aggregation and anti-repetition design

The platform should stop treating this as “dedup only”. The actual requirement is:

- remove literal duplicates
- merge near-duplicate materials into one usable evidence cluster
- keep enough related materials to enrich one article or case study
- block repeated topic selection across recent outputs

The current URL-only dedup must be replaced by three levels of code-driven control.

### Level 1: canonical URL dedup

- normalize trailing slash, fragments, tracking params, known redirect patterns
- one `normalized_url` maps to one material record lineage

### Level 2: content hash dedup

- hash normalized main text
- if different URLs produce near-identical text, keep one primary record and mark others duplicate

### Level 3: semantic cluster aggregation

- cluster materials by title semantics, source entities, time window, key signals, tag outputs, and content similarity
- each cluster gets one `cluster_id`
- each cluster stores:
  - one `primary_material`
  - zero or more `supporting_materials`
  - a `cluster_summary`
  - `cluster_signals`
- article and case pipelines select clusters, not raw URLs

This is the missing layer that prevents “同一话题换个来源再写一次”, while still allowing a business pipeline to combine several similar materials into one richer output.

## Editorial fit scoring

Relevance alone is not enough. A material can be AI-related and still be a bad fit for this project.

The platform should add `editorial_fit_score` as a first-class field during curation and dataset building.

High-fit materials are those that can be written into:

- enterprise organizational change
- workflow redesign
- human-AI work split
- adoption resistance and trust
- governance, permissions, and ownership boundaries
- performance, roles, incentives, and management shifts
- mechanism-level interpretation rather than news recap

Low-fit or reject materials are those that are mostly:

- pricing strategy
- packaging changes
- feature launches
- partnership or funding news
- product updates without organizational implications
- business strategy moves that do not clearly connect to process, org, or management change

This directly addresses cases like a platform pricing story that is AI-related but does not say anything meaningful about how enterprises reorganize around AI.

## Topic selection rule for business strategy stories

Business strategy stories are not excluded by category. They are excluded unless they can be translated into enterprise organizational meaning.

For example, a pricing or packaging change only belongs in the writing pool if the material supports analysis about at least one of:

- procurement and budget ownership changes
- permissioning and access control shifts
- team-level tool adoption constraints
- collaboration boundary changes
- Shadow AI governance
- organizational standardization pressure

If that bridge cannot be made from the source material and supporting cluster evidence, the material must not become a primary topic candidate.

## Topic memory and novelty scoring

Topic repetition should be solved before writing, not after publishing.

Maintain `platform/state/topic_memory.json` with recent outputs:

```json
{
  "recent_outputs": [
    {
      "output_id": "article_2026-06-03_xxx",
      "business_type": "daily_article",
      "title": "为什么团队买了AI工具却没用起来",
      "published_at": "2026-06-03T06:45:00+08:00",
      "cluster_ids": ["cluster_019", "cluster_023"],
      "topic_labels": ["员工采纳", "组织变革"],
      "novelty_signature": ["tool adoption", "employee resistance", "workflow redesign"]
    }
  ]
}
```

Before topic selection:

- exclude clusters used in the last N days
- penalize topic labels overused in the last N outputs
- penalize repeated entity combinations
- penalize business-strategy-only topics that lack organizational signals
- require a minimum novelty score before a topic can be selected

Recommended defaults:

- hard block same `cluster_id` for 14 days
- soft block same first-order topic combination for 7 outputs
- soft block same named entity pair for 5 outputs
- hard block any topic candidate whose `editorial_fit_score` is below the primary-topic threshold

This should be deterministic code, not an instruction hidden inside a writing prompt.

## Dataset rules

`article_pool.json` and `case_pool.json` should not be “all pass materials dumped together”.
They should already be business-ready.

### `article_pool.json`

Contains only materials that pass:

- relevance
- minimum content quality
- language and policy checks
- not recently used
- not duplicate within cluster
- suitable for narrative synthesis

Also contains scoring fields:

- `editorial_fit_score`
- `novelty_score`
- `narrative_score`
- `evidence_score`
- `freshness_score`

### `case_pool.json`

Contains only materials that pass:

- relevance
- case-study suitability
- enough operational detail
- measurable business context or execution details
- cross-reference potential with other materials in the same cluster

Also contains:

- `editorial_fit_score`
- `case_depth_score`
- `verification_candidates`
- `execution_detail_score`

## 30-day cleanup policy

Storage must stay file-based, but not accumulate unbounded context.

Cleanup policy:

- delete `platform/ingest/raw/<date>/` older than 30 days
- delete `platform/normalize/<date>/` older than 30 days
- delete `platform/curate/<date>/` older than 30 days
- delete failed or superseded job log directories older than 30 days
- keep `platform/state/*.json` as compact long-lived summaries
- keep final published outputs by default unless explicitly archived elsewhere

Additionally:

- compact topic memory to only the most recent 60 outputs
- compact publish index to only fields needed for dedup and audit
- rebuild derived caches instead of treating them as source of truth

## Migration strategy

The user explicitly chose to drop historical baggage. So migration should be minimal.

Plan:

1. Freeze old scripts as legacy execution path.
2. Create a new `platform/` package and a new `platform_cli.py` entrypoint.
3. Route cron to new jobs only after platform parity is verified.
4. Keep old data outside the new pipeline and archive it.
5. Do not attempt compatibility with `all_urls.tsv`, `tmp/`, or old tag stores inside the new runtime.

This avoids spending time on adapter code that preserves flawed assumptions.

## Failure handling rules

To remove probability-driven behavior, the platform should adopt strict failure rules:

- if source collection fails, record failure and stop that branch
- if normalization produces zero materials unexpectedly, mark the job failed
- if curation fails, keep the last completed dataset snapshot; do not emit partial curated output as successful
- if business writing fails, retain the dataset and selection artifacts for exact rerun
- if publishing fails, do not lose the generated output directory
- every external API call records request metadata and output artifact references for debugging

## Testing strategy

The redesign should be verified at three levels.

### Unit tests

- URL normalization
- content hash dedup
- cluster assignment
- novelty scoring
- cleanup retention logic
- job resume behavior

### Integration tests

- `collect-daily` end-to-end on fixture inputs
- `article-daily` using a frozen curated dataset
- `case-daily` using a frozen curated dataset
- rerun after a simulated mid-job failure

### Fixture regression tests

- duplicate URLs from different sources collapse into one cluster
- same topic with different sources is blocked by novelty rules
- materials older than 30 days are removed by cleanup without breaking state

## Rollout plan

1. Build the new platform side-by-side with legacy scripts.
2. First replace collection and curation, but keep business output unpublished.
3. Compare selected materials and output quality for several days.
4. Switch `daily_article` to the new platform.
5. Switch `case_study` to the new platform.
6. Remove legacy cron entries and old stateful scripts from active use.

## Explicit non-goals

- no database introduction
- no attempt to preserve compatibility with old TSV-based state
- no generic workflow engine beyond what this repository needs
- no reintroduction of agent-driven orchestration

## Recommended next implementation boundary

Implement the shared platform first, then port both business flows onto it.

The first milestone should produce:

- new directory structure
- job state management
- ingest to curated pipeline
- dataset builders
- cleanup command

Only after that should the writing and publishing flows be migrated.
